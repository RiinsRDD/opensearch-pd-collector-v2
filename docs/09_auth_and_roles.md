# 9. Авторизация и Роли (Auth & Roles)

## Текущее состояние

Авторизация реализована как **mock** — без реальной проверки учётных данных и JWT-токенов.

### Файл: `app/api/endpoints/auth.py`

| Метод | Путь | Ответ |
|-------|------|-------|
| `POST` | `/api/v1/auth/login` | `{"access_token": "mock_token", "token_type": "bearer"}` |
| `GET` | `/api/v1/auth/me` | `{"username": "admin", "role": "admin"}` |

> **Нет реальной JWT-верификации**, ролевой проверки или хранения пользователей в БД.

## Целевая архитектура (спецификация)

### Ролевая модель

| Роль | Права |
|------|-------|
| **Viewer** | Просмотр дашбордов, дерева, `raw_document`. Нет действий |
| **Analyst** | Всё от Viewer + тегирование, обновление примеров, создание задач Jira, добавление `RegexRule` |
| **Admin** | Полный доступ: глобальные настройки, управление `IndexOwners`, ручной запуск сканирования, удаление из кэша |

### Персональные ключи Jira

Каждый пользователь должен хранить свой **Jira API Token** для создания задач от своего имени (не от сервисного аккаунта).

### Механизм JWT-сессий (план реализации)

1. Пользователь вводит логин/пароль на Frontend.
2. `POST /api/v1/auth/login` → генерация `access_token` JWT с `user_id` и `role`.
3. Axios прокидывает `Authorization: Bearer <token>` во все запросы.
4. DI-зависимости FastAPI (`get_current_user`, `require_admin`) проверяют роль.

### Зависимости для реализации

В `requirements.txt` уже включены:

- `python-jose[cryptography]` — генерация/верификация JWT
- `passlib[bcrypt]` — хеширование паролей
