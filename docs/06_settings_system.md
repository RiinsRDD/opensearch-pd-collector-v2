# 6. Система настроек (Settings System)

Все параметры хранятся в PostgreSQL в таблицах `system_settings`, `regex_rules` и `status_settings`, что позволяет редактировать их через Frontend без перезапуска контейнера.

## Глобальные настройки (`SystemSetting`)

Ключи настроек, хранящихся в таблице `system_settings`:

| Ключ | Тип | Описание |
|------|-----|----------|
| `EXAMPLES_COUNT` | int | Количество примеров для каждого `cache_key` |
| `SCAN_INTERVAL_HOURS` | int | Интервал глобального сканирования |
| `BATCH_SIZE` | int | Размер пачки документов при скачивании (по умолчанию 10 000) |
| `IS_{TYPE}` (например `IS_PHONE`, `IS_INN`) | bool | Динамические флаги включения/отключения соответствующих детекторов ПДн |
| `jira_{field}` | string/int | Настройки для интеграции с Jira (base URL, ключи проектов, custom fields, приоритет и т.д.) |

Настройки выдаются и обновляются через API:

- `GET /api/v1/settings/global`
- `POST /api/v1/settings/global`

## Статусы (`StatusSetting`)

Настраиваемые статусы с цветами:

| ID | Метка | Цвет по умолчанию |
|----|-------|-------------------|
| `new` | New | `#ef4444` |
| `confirmed` | Confirmed | `#3b82f6` |
| `done` | Done | `#10b981` |
| `false_positive` | False Positive | `#eab308` |
| `unverified` | Unverified | `#94a3b8` |

API:

- `GET /api/v1/settings/statuses`
- `POST /api/v1/settings/statuses`

## Управление правилами и регулярками (`RegexRule`)

Таблица `regex_rules` содержит типы записей:

| `rule_type` | Назначение |
|-------------|------------|
| `regex` | Основное регулярное выражение для типа ПДн (системного или пользовательского) |
| `exclude_pattern` | Глобальное исключение по полному совпадению или подстроке для значений |
| `prefix_exclude` | Исключение по префиксу значения |
| `suffix_exclude` | Исключение по суффиксу значения |
| `exclude_key` | Исключение конкретного ключа везде |
| `full_path_key.exclude` | Глобальное исключение по конкретному пути JSON |
| `invalid_def_code` | Невалидный DEF-код для телефонов |
| `exclude_index_pattern` | Исключение индекса по точному совпадению или подстроке (системное правило |
| `exclude_index_regex` | Исключение индекса по регулярному выражению (системное правило) |
| `include_index_regex` | Включение индекса по регулярному выражению (системное правило) |
| `mail_service_name` | Почтовые домены считающиеся за валидный email |
| `unknown_mail_service_part` | Неизвестные/специфичные почтовые домены для аналитики |
| `card_bank_bin_4` | 4-х значные БИНы банковских карт (в основном РФ) |
| `surn_end_cis` | Типичные окончания фамилий СНГ для ФИО |
| `surn_end_world` | Западные/Европейские окончания фамилий для ФИО |
| `patron_end` | Окончания отчеств для ФИО |
| `fio_special_marker` | Спецмаркеры ФИО (оглы, кызы, фон, ван) |

API для управления типами ПДн и их основными регулярками (`rule_type="regex"`):

- `GET /api/v1/settings/pdn-types`
- `GET /api/v1/settings/pdn-types/list`
- `POST /api/v1/settings/pdn-types`
- `PUT /api/v1/settings/pdn-types/{rule_id}`
- `DELETE /api/v1/settings/pdn-types/{rule_id}`

> **Примечание по словарям ПДн:** Для оптимизации алгоритмов поиска все строковые словари ПДн (кроме `unknown_mail_service_part`) при сохранении через Frontend автоматически переводятся в нижний регистр, сортируются по алфавиту и очищаются от дубликатов. Детекторы (например в `app/services/detectors.py`) также используют `lower()` для сравнения, в то время как в базу `PDNFinding` оригинальный регистр найденной строки сохраняется неизменным.

## Индивидуальные настройки индексов (`IndexOwner` & `IndexKeyExclusion`)

Через API `/api/v1/settings/index-owners` можно настроить для конкретного индекса (в `IndexOwner`) маппинг для автоматического заведения задач в Jira:

- ФИО ответственного (`fio`) — подставляется в `assignee`
- URL IT-системы (`cmdb_url`)
- Идентификатор Тех. долга (`tech_debt_id`)

Также индексы могут иметь собственные исключения по путям (`IndexKeyExclusion`):

- Через вкладку "Исключения индексов" в UI или API (`/api/v1/settings/exclusions/index`) можно добавлять полные пути ключей (например `kubernetes.namespace.container`), которые будут игнорироваться для этого индекса по выбранному типу ПДн.

## Дополнительные поля разграничения (Scan Fields)

Конфигурации дополнительных полей хранятся в таблице `scan_field_configs` и управляются через API `/api/v1/settings/scan-fields`.
Они позволяют извлекать из логов специфичные поля (например, `NameOfMicroService` или `kubernetes.container.name`) и включать их в `cache_key`. Это предотвращает смешивание находок ПДн из разных микросервисов в один инцидент. Обязательные поля помечаются флагом `is_required=True` и не могут быть удалены через UI.

## Управление тегами

Глобальное удаление тега: `DELETE /api/v1/settings/tags/{tag_name}` — каскадно удаляет тег со всех `PatternTagLink`.

## Цикл применения настроек

Детекторы оперируют данными из БД. При следующем тике `APScheduler` сервис обновляет правила:

1. Изменение параметра в UI → запись в `system_settings`.
2. При следующем запуске сканера → новые правила применяются.
3. Ранее сохранённые примеры остаются без изменений.
